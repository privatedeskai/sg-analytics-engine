# SG Analytics Engine — Журнал архитектурных решений

> Каждое важное решение фиксируется здесь. Это предотвращает повторное обсуждение и сохраняет контекст между сессиями.

---

## 2026-05-04 — Выбор AI модели для аналитического агента

**Решение:** Kimi K2.6 через Gonka AI (основная) + Claude API (Vision + финальный текст)

**Причина:**
- Kimi K2.6 в 4 раза дешевле Claude Sonnet при сопоставимом качестве для кода
- Tool invocation success rate 96% — достаточно для итерационного цикла
- Галлюцинации на данных устранены архитектурно через E2B
- Claude API остаётся для Vision (Kimi слабый, #26 из 115) и финального текста

**Альтернативы отклонены:**
- CoPaw LoRA на Fireworks — требует DevOps, GPU деплой, разработчик
- Только Claude API — дороже в 4 раза при масштабе
- DeepSeek — серверы в Китае, риск для US/CA рынка

## 2026-05-04 — Execution engine

**Решение:** Judge0 CE (ce.judge0.com) — без ключа, без регистрации

**Причина:**
- E2B SDK несовместим с Cloudflare Workers (ESM/gRPC проблема) — не работает
- Piston API закрыт с 15 февраля 2026 — не работает
- Judge0 CE: бесплатно, без ключа, без карты, pandas/numpy включены, чистый HTTP REST
- Полностью совместим с Cloudflare Workers

**Альтернативы отклонены:**
- E2B SDK — несовместим с CF Workers (ESM конфликт chalk зависимости)
- Piston API — закрыт для публичного доступа с 15.02.2026
- RapidAPI/Judge0 — требует карту даже на бесплатном плане
- Локальный Python — небезопасно для мультипользователей

## 2026-05-04 — Оркестратор

**Решение:** Cloudflare Durable Objects для длинных запросов

**Причина:**
- Обычный Worker лимит 30 сек — 10 итераций × 3-5 сек = 30-50 сек, граничное значение
- Durable Objects нет лимита времени + stateful (хранит контекст итераций)
- Уже в инфраструктуре Cloudflare — не новый сервис

**Альтернативы отклонены:**
- Modal.com — нужен Python разработчик для деплоя
- FastAPI сервер — нужна отдельная инфраструктура

## 2026-05-04 — Стратегия запуска

**Решение:** Сначала универсальный CSV Analyst, потом специализация

**Причина:**
- Нет реальных пользователей — нет данных для выбора специализации
- 3 недели до рабочего MVP vs 8 недель до специализированного продукта
- Данные от бета-пользователей покажут куда специализироваться

**Альтернативы отклонены:**
- Сразу SG Competitive Intelligence — рынок переполнен без дифференциации
- Сразу MarketLens — не валидирован спрос

## 2026-05-04 — Данные через Kimi (риск Китай)

**Решение:** Использовать DeepInfra как провайдер Kimi K2.6 (серверы в США)

**Причина:**
- Kimi K2.6 открытые веса (MIT лицензия) — можно запускать на любой инфраструктуре
- DeepInfra хостит K2.6 на серверах в США
- Данные пользователей физически остаются в США

**Fallback:** Claude API если DeepInfra нестабилен — замена одной строки в конфиге

## 2026-05-05 — Временные решения (ВЕРНУТЬ ПОСЛЕ СТАБИЛИЗАЦИИ)

> ⚠️ Это раздел активных технических долгов. Проверять в начале каждой сессии.

### ДОЛГ 1: Claude API вместо Kimi K2.6
**Текущее состояние:** kimi.ts использует claude-sonnet-4-5 через Anthropic API
**Причина:** Нулевой баланс на DeepInfra
**Что вернуть:** Пополнить баланс на DeepInfra (~$10) → в kimi.ts изменить:
- baseUrl: 'https://api.deepinfra.com/v1/openai'
- model: 'moonshotai/Kimi-K2-Instruct'
**Влияние:** В 20 раз дороже при масштабе пока не исправлено

### ДОЛГ 2: MAX_ITERATIONS = 3
**Текущее состояние:** wrangler.toml и orchestrator.ts ограничивают до 3 итераций
**Причина:** Отладка таймаутов Durable Object
**Что вернуть:** После подтверждения стабильной работы:
- wrangler.toml: MAX_ITERATIONS = "10"
- orchestrator.ts: убрать `Math.min(req.maxIterations || 3, 3)` → вернуть `req.maxIterations || 10`
**Влияние:** Качество анализа ниже чем планировалось (3 итерации вместо 10)

### ДОЛГ 3: Таймауты в orchestrator.ts
**Текущее состояние:** withTimeout() на каждом шаге (20 сек Kimi, 15 сек Judge0, 25 сек summary)
**Причина:** Временная мера для предотвращения зависания
**Статус:** Возможно оставить как есть — это хорошая практика
**Решение:** После стабилизации оценить нужно ли увеличить лимиты

## 2026-05-05 — Хранение статуса сессий

**Решение:** Статус пишется в Cloudflare KV (не в DO internal storage)

**Причина:**
- DO internal storage недоступен из index.ts при polling /result
- KV читается из любого Worker без создания нового DO экземпляра
- KV TTL 24 часа — достаточно для сессий

**Ключ:** session:{sessionId}
